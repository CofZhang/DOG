# 调试记录 2026-05-27 — 电机掉线检测修复

## 1. 问题背景

上次调试完成了LED指示系统和蜂鸣器报警的基础功能后，在实际连接电机测试时发现掉线检测逻辑存在多个问题，本次进行系统性修复。

---

## 2. 问题一：连接电机后仍显示掉线（SPI侧电机4-12）

### 现象

连接全部12个电机，Keil Watch 窗口可以读到 `g_motor_feedback` 有数据，但 LED6/LED7/LED8 仍然报掉线。LED5（电机1-3，主控直连CAN）正常。

### 根本原因

`u_spi.c` 的 `SPI_TxCpltCallback()` 中，从机反馈解包后**从未给 `timestamp` 字段赋值**：

```c
// 原代码：只解包数据，timestamp 永远是0
Motor_UnpackFeedbackData(motor_data_ptr + i * 8, &g_slave_feedback[i]);
```

而掉线检测依赖 `timestamp` 是否更新来判断是否收到反馈。主控侧（电机1-3）的 `timestamp` 在 `fdcan_handler.c` 的接收中断里赋值，但从机侧经 SPI 转发后没有人更新 `timestamp`，导致始终判定为掉线。

### 第一步修复（不完整）

在解包后统一打时间戳：

```c
uint32_t now = HAL_GetTick();
for (int i = 0; i < SPI_PKG_MOTOR_CNT; i++) {
    Motor_UnpackFeedbackData(motor_data_ptr + i * 8, &g_slave_feedback[i]);
    g_slave_feedback[i].timestamp = now;
}
```

**问题**：SPI 是全双工，从机发回的是固定9个电机的数据包，即使某个电机没有反馈，从机也会把上一轮的旧数据填进去。主控无法区分"本轮有新反馈"和"旧数据重发"，导致掉线检测失效。

---

## 3. 问题二：从机无法区分单个电机是否有新反馈

### 解决方案：在 SPI 反馈包的 reserved 字段加入 valid_mask

`SPIFeedbackPacket_t` 中有 `reserved[2]` 两个保留字节，利用这两个字节传递9个电机的有效标志位：

- `reserved[0]`：bit0~bit7 对应电机4~11
- `reserved[1]`：bit0 对应电机12

**从机改动（`DOG_Slaver`）：**

`slave_controller.h` — `SlaveController_t` 新增字段：
```c
volatile uint16_t motorValidMask;  // 本轮收到反馈的电机位掩码
```

`slave_controller.c` — FDCAN收到反馈时置位：
```c
g_slaveCtrl.motorValidMask |= (1u << i);
```

`slave_controller.c` — 打包时写入 reserved，打包完清零：
```c
feedback->reserved[0] = (uint8_t)(g_slaveCtrl.motorValidMask & 0xFF);
feedback->reserved[1] = (uint8_t)((g_slaveCtrl.motorValidMask >> 8) & 0x01);
// ...
g_slaveCtrl.motorValidMask = 0;  // 清零，为下一轮准备
```

**主控改动（`u_spi.c`）：**

```c
uint16_t valid_mask = (uint16_t)g_spi_rx_buffer[4]
                    | ((uint16_t)(g_spi_rx_buffer[5] & 0x01) << 8);
uint32_t now = HAL_GetTick();
for (int i = 0; i < SPI_PKG_MOTOR_CNT; i++) {
    Motor_UnpackFeedbackData(motor_data_ptr + i * 8, &g_slave_feedback[i]);
    if (valid_mask & (1u << i)) {
        g_slave_feedback[i].timestamp = now;  // 只给本轮有效的电机打时间戳
    }
}
```

## 4. 问题三：只连部分电机时，未连接的电机不报掉线

### 现象

只连接 ID4/5/6，理论上 LED5（电机1-3）、LED7（电机7-9）、LED8（电机10-12）应该报掉线，但蜂鸣器不响，LED 全部不亮。

### 根本原因

上一次调试为了解决"上电无电机时蜂鸣器误报"，加入了 `g_motor_ever_online` 标志：

```c
// 从未收到过反馈 → 不计入掉线，忽略
} else if (g_motor_ever_online[i]) {
    // 累积掉线计数
}
```

这导致从未上线的电机永远不会触发掉线报警，与"运行中未连接的电机应该报警"的需求矛盾。

### 解决方案：宽限期机制

系统进入 `SYS_STATE_RUNNING`（收到第一包USB数据）后，给3秒宽限期：

- **宽限期内**：从未在线的电机不报警（避免上电瞬间误报）
- **宽限期结束后**：无论是否曾经在线，只要本轮没有收到反馈就累积掉线计数

```c
#define MOTOR_ONLINE_GRACE_MS  3000   // 宽限期3秒

// 进入RUNNING时记录时间（仅第一次）
if (g_system_state != SYS_STATE_RUNNING) {
    g_running_start_tick = HAL_GetTick();
}
g_system_state = SYS_STATE_RUNNING;

// 掉线检测
uint8_t grace_over = (HAL_GetTick() - g_running_start_tick) >= MOTOR_ONLINE_GRACE_MS;

if (ts != 0 && ts != g_last_feedback_ts[i]) {
    // 正常
} else if (g_motor_ever_online[i] || grace_over) {
    // 累积掉线计数
}
// 宽限期内且从未在线：不计入掉线
```

---

## 5. 最终掉线检测状态机

```
上位机开始发数据（进入RUNNING）
  │
  ├─ 前3秒（宽限期）
  │    ├─ 收到反馈 → ever_online=1，正常
  │    └─ 没收到反馈 → 不报警
  │
  └─ 3秒后（宽限期结束）
       ├─ 收到反馈 → 正常，计数清零
       └─ 没收到反馈 → 计数+1 → 超过20次 → 掉线标志=1 → LED闪烁 + 蜂鸣器报警
```
