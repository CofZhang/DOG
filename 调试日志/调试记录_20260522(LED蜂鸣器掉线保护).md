# 调试记录 2026-05-22 — LED指示系统 / 蜂鸣器 / 电机掉线保护

## 1. 问题背景

本次调试涉及三个相关模块的修复与重构：

- 蜂鸣器驱动频率错误
- LED指示灯功能混乱（LED编号与实际功能不对应，部分LED从未触发）
- 电机掉线保护误报（上电无电机时蜂鸣器持续鸣叫）

## 2. 蜂鸣器频率修复

### 问题

`led.c` 中 `Buzzer_Beep()` 的计数时钟写错，导致实际频率只有期望的一半：

```c
// 错误：
uint32_t tmr_clk = 1000000;  // 误以为是1MHz

// 正确：
uint32_t tmr_clk = 500000;   // TIM17: 120MHz APB2 / (239+1) = 500kHz
```

### 原因分析

TIM17 配置：`Prescaler = 239`，APB2 时钟 = 120MHz。

```
计数时钟 = 120MHz / (239 + 1) = 500kHz
```

原代码填写 1MHz，导致 `BEEP_On()` 调用 `Buzzer_Beep(1000)` 实际输出 500Hz，而非 1000Hz。

### 修复文件

`DOG_Master/Core/Src/led.c`

## 3. LED指示系统重构

### 问题诊断

对照 `main.h` 引脚定义和 `Led_indicator.c` 实现，发现以下错误：

| 期望功能 | 期望LED | 实际代码驱动的LED | 问题 |
|---|---|---|---|
| USB数据接收 | LED2 | LED2（逻辑正确，但**从未触发**） | `Protocol_ProcessRxData()` 里没有调用触发函数 |
| CAN发送 | LED3 | LED4（`LED_IND_FDCAN_TX` → `LED4_On()`） | LED编号对调 |
| SPI发送 | LED4 | LED3（`LED_IND_SPI_TX` → `LED3_On()`） | LED编号对调 |
| 电机掉线指示 | LED5~LED8 | 无 | 掉线只触发蜂鸣器，没有LED指示 |

### 修复方案

**重新定义枚举**（`Led_indicator.h`）：

```c
typedef enum {
    LED_IND_SYSTEM_HEARTBEAT = 0,   // LED1：系统心跳
    LED_IND_USB_RX,                 // LED2：USB数据接收
    LED_IND_FDCAN_TX,               // LED3：CAN发送
    LED_IND_SPI_TX,                 // LED4：SPI发送
    LED_IND_COUNT
} LED_IndicatorType;
```

删除了不再使用的 `LED_IND_USB_TX`、`LED_IND_MOTOR_FEEDBACK`、`LED_IND_USB_ERROR`、`LED_IND_SPI_ERROR`、`LED_IND_FDCAN_ERROR`、`LED_IND_CRITICAL_ERROR`。

**修复 LED3/LED4 映射**（`Led_indicator.c`）：

```c
// LED3 → CAN发送（FDCAN_TX）
case LED_IND_FDCAN_TX: LED3_On(); ...

// LED4 → SPI发送（SPI_TX）
case LED_IND_SPI_TX:   LED4_On(); ...
```

**补充 USB 接收触发**（`usbcol.c`，`Protocol_ProcessRxData()`）：

```c
if (processed > 0) {
    LED_Indicator_Trigger(LED_IND_USB_RX);  // 触发LED2闪烁
}
```

**新增掉线分组 LED 指示**（`system_control.c`，`System_CheckMotorOffline()`）：

```c
// LED5：电机1-3（索引0-2）
uint8_t offline_1_3 = g_motor_offline[0] || g_motor_offline[1] || g_motor_offline[2];
if (offline_1_3) { LED5_On(); } else { LED5_Off(); }

// LED6：电机4-6（索引3-5）
// LED7：电机7-9（索引6-8）
// LED8：电机10-12（索引9-11）
```

**清理 `fdcan_handler.c`**：删除了对已移除枚举 `LED_IND_MOTOR_FEEDBACK`、`LED_IND_FDCAN_ERROR` 的调用。

### 最终 LED 功能对照

| LED | 引脚 | 功能 | 行为 |
|-----|------|------|------|
| LED1 | PB6 | 系统心跳 | 1Hz 闪烁 |
| LED2 | PB7 | USB数据接收 | 每收到一包亮 50ms |
| LED3 | PB8 | CAN发送 | 每发送成功亮 30ms |
| LED4 | PE0 | SPI发送 | 每发送成功亮 30ms |
| LED5 | PE1 | 电机1-3掉线 | 掉线常亮 |
| LED6 | PE2 | 电机4-6掉线 | 掉线常亮 |
| LED7 | PE3 | 电机7-9掉线 | 掉线常亮 |
| LED8 | PE4 | 电机10-12掉线 | 掉线常亮 |

## 4. 掉线保护误报修复

### 问题

上电后蜂鸣器持续鸣叫，即使没有连接电机、没有发送 USB 数据。

### 根本原因

掉线检测逻辑：反馈时间戳为 0 或未更新 → 计数器递增 → 超过阈值20次 → 判定掉线 → 触发蜂鸣器。

上电时没有电机连接，所有12个电机的时间戳永远为0，掉线计数器在上电采样阶段（`System_BootCalibSample`）就快速累积到阈值，导致误报。

### 修复

在 `System_SendFeedback()` 中加入状态判断，**只有系统进入 `SYS_STATE_RUNNING`（即收到过至少一次 USB 控制数据）后，才启用掉线检测**：

```c
// system_control.c - System_SendFeedback()
if (g_system_state == SYS_STATE_RUNNING) {
    System_CheckMotorOffline();
}
```

状态转换逻辑：

```
上电 → SYS_STATE_IDLE（掉线检测不生效）
  ↓ 收到第一包 USB 数据
SYS_STATE_RUNNING（掉线检测开始生效）
```

## 5. 掉线报警行为调整

将原来的"持续鸣叫"改为"0.5s 间歇鸣叫"，使用 `Tim_IsElapsed()` 非阻塞实现：

```c
if (any_offline) {
    static uint32_t beep_tick = 0;
    static uint8_t  beep_state = 0;
    if (Tim_IsElapsed(&beep_tick, 500)) {
        beep_state = !beep_state;
        if (beep_state) { BEEP_On(); } else { BEEP_Off(); }
    }
} else {
    BEEP_Off();
}
```

## 6. 新增非阻塞延迟模块（tim_delay）

### 背景

主循环中不能使用 `HAL_Delay()`，否则阻塞所有任务。新建 `tim_delay.c / tim_delay.h`。

### 接口

```c
void    Tim_Delay(uint32_t ms);                              // 阻塞式，仅用于初始化
uint8_t Tim_IsElapsed(uint32_t *last_tick, uint32_t ms);    // 非阻塞，主循环定时触发
uint32_t Tim_GetMs(void);                                    // 获取当前毫秒时间戳
```

底层使用 `HAL_GetTick()`（SysTick，1ms精度），不占用额外硬件定时器。

### 注意

`tim_delay.c` 需要手动在 Keil 工程中添加（右键 `Application/User/Core` → Add Existing Files），否则每次 Keil 保存工程会将其从 `.uvprojx` 中移除。
