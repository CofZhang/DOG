## 二、从控 SPIPacket_t 结构体与主控不匹配

### 问题描述
从控的 `SPIPacket_t` 为 88 字节（双帧头帧尾、16位校验和），主控发送的是 80 字节格式。

### 主控实际格式（80字节）
```
[header(1)][cmdType(1)][motorCnt(1)][sequence(1)][reserved(2)][9×8 motor data(72)][checksum(1)][footer(1)]
```

### 修复
重写 `motor_protocol.h` 中的 `SPIPacket_t`：
```c
typedef struct {
    uint8_t header;          // 0xAA
    uint8_t cmdType;         // 0x10
    uint8_t motorCnt;        // 9
    uint8_t sequence;
    uint8_t reserved[2];
    uint8_t motorData[9][8]; // 72字节
    uint8_t checksum;        // XOR(Byte 0~77)
    uint8_t footer;          // 0x55
} __attribute__((packed)) SPIPacket_t;  // 80字节
```

## 五、FDCAN2/3 过滤器配置错误

### 问题描述
CubeMX 生成的 `fdcan.c` 中 FDCAN2 和 FDCAN3 的 `StdFiltersNbr = 0`，导致无法配置标准帧过滤器，`HAL_FDCAN_ConfigFilter()` 会失败。

### 修复

## 六、主控 SPI DMA 对齐问题（HAL_BUSY）

### 问题描述
主控调用 `HAL_SPI_Transmit_DMA()` 返回 `HAL_BUSY`。

### 根本原因
主控 SPI 配置为 16 位模式（`SPI_DATASIZE_16BIT`），但 DMA 配置为字节对齐（`DMA_PDATAALIGN_BYTE` / `DMA_MDATAALIGN_BYTE`），HAL 内部检查不通过。

### 修复
在主控 CubeMX 中将 SPI1 的 DMA 配置改为：
- PeriphDataAlignment = **HalfWord**
- MemDataAlignment = **HalfWord**

DMA 传输长度单位变为半字：`sizeof(SPIPacket_t) / 2 = 40`

## 七、从控 SPI 接收永远不触发（spiRxComplete 始终为 0）

### 现象
- 主控 `HAL_SPI_Transmit_DMA()` 返回 `HAL_OK`（发送成功）
- 从控 `g_slaveCtrl.spiRxComplete` 始终为 0
- `HAL_SPI_RxCpltCallback` 从未被调用
- 超时保护 `if ((HAL_GetTick() - g_slaveCtrl.lastRxTime) > 1000)` 一直触发

### 调试方法

#### 断点位置
```
slave_controller.c - SlaveController_Process()
在 if (g_slaveCtrl.spiRxComplete) 处打断点，观察是否进入
在超时保护 if ((HAL_GetTick() - g_slaveCtrl.lastRxTime) > 1000) 处打断点
```

#### 观测内容（Keil Watch 窗口）
```
slave_controller\g_slaveCtrl.spiRxComplete    ← 应为1，实际始终为0
slave_controller\g_slaveCtrl.spiTxComplete    ← 应为1，实际始终为0
slave_controller\g_slaveCtrl.spiRxPacket      ← 查看是否有数据写入
hspi1.State                                    ← 查看SPI状态
hspi1.hdmarx->Instance->CNDTR                  ← DMA剩余传输计数
```

#### 排查过程
1. ✅ DMA NVIC 已使能（dma.c 中 DMA1_Channel1/2 中断已开启）
2. ✅ DMA IRQ Handler 存在（stm32g4xx_it.c 中调用 HAL_DMA_IRQHandler）
3. ✅ HAL_SPI_RxCpltCallback 已定义（slave_controller.c）
4. ✅ 初始化顺序正确：MX_DMA_Init() → MX_SPI1_Init() → SlaveController_Init()
5. ✅ SPI 参数匹配：主从都是 16bit、CPOL=LOW、CPHA=1EDGE、MSB First

### 根本原因：主控 NSSPMode = SPI_NSS_PULSE_ENABLE

主控 `spi.c` 第54行：
```c
hspi1.Init.NSSPMode = SPI_NSS_PULSE_ENABLE;
```

此设置导致主控在**每个16位数据帧之间**产生 NSS 脉冲（NSS 拉高再拉低）。从控配置了 `NSS = SPI_NSS_HARD_INPUT`，每次 NSS 上升沿都被解释为传输结束，SPI 外设被复位。

**结果**：主控发送40个半字，但从控每收1个半字就认为传输结束，DMA 永远无法收满40个半字，`HAL_SPI_RxCpltCallback` 永远不会触发。

### 修复
在主控 CubeMX 中：
1. 打开 SPI1 配置
2. 找到 NSSP Mode（NSS Signal Type）
3. 设为 **Disable**

生成代码后第54行变为：
```c
hspi1.Init.NSSPMode = SPI_NSS_PULSE_DISABLE;
```

从控不需要修改。NSS 在整个 DMA 传输期间保持低电平，从控 DMA 才能正常接收完整的80字节。

## 九、调试经验总结

1. **SPI 从机 + NSS 硬件模式时，主机必须关闭 NSSP 脉冲**，否则每个数据帧之间的 NSS 脉冲会中断从机的连续接收。

2. **SPI 16位模式下 DMA 必须配置为 HalfWord 对齐**，否则 HAL 库内部检查会返回 HAL_BUSY。

3. **DMA 传输长度单位是数据宽度**，不是字节数。80字节 ÷ 2 = 40 半字。

4. **Keil 观测 static 变量**需要加模块限定符：`模块名\变量名`（如 `slave_controller\g_slaveCtrl`）。

5. **CubeMX 生成的代码**中 FDCAN 的 StdFiltersNbr 默认为 0，如果需要配置过滤器必须手动改为 ≥1。

## 十、NSSP 改了依然只收到 12/40 半字（最终修复）

### 现象
- 主控 `HAL_SPI_TxCpltCallback` 正常触发，`hspi1.ErrorCode == 0`
- 从控 `hdma_spi1_rx.Instance->CNDTR == 28`（即只收到 40-28=12 个半字 = 24 字节）
- 从控 `hspi1.State == 0x04`（HAL_SPI_STATE_BUSY_RX，DMA 还在等待剩余数据）
- 降低主控 SPI 时钟（Prescaler 8 → 32）无效
- NSSP Mode 已确认是 DISABLE

### 根本原因：全双工 SPI 的 RX FIFO 阻塞
STM32H7 的 SPI 是全双工外设。主控调用 `HAL_SPI_Transmit_DMA()` 时，硬件上 MISO 线依然在接收从控输出的数据，这些数据会进入主控的 RX FIFO。由于主控没有启用 RX DMA、也没有读取 RX FIFO，RX FIFO 很快被填满（FIFO 深度对应约 12 个半字），硬件触发 FIFO overrun 并自动暂停 SCK 时钟。从控因此只收到 12 个半字就卡住。

### 修复：主从两端都改用 `HAL_SPI_TransmitReceive_DMA`
主控和从控都启动全双工 DMA，TX 发控制指令、RX 同时收反馈包（或反之），这样主控的 RX FIFO 会被 DMA 持续排空，SCK 就能一直走完 40 个半字。

#### 主控 `u_spi.c` 修改要点
```c
static uint8_t g_spi_tx_buffer[SPI_PKG_TOTAL_LEN];
static uint8_t g_spi_rx_buffer[SPI_PKG_TOTAL_LEN];  // 新增：接收从控反馈

// SPI_SendMotorControl 内：
HAL_SPI_TransmitReceive_DMA(&hspi1, g_spi_tx_buffer, g_spi_rx_buffer, transfer_len);

// 回调从 HAL_SPI_TxCpltCallback 改为：
void HAL_SPI_TxRxCpltCallback(SPI_HandleTypeDef *hspi) { ... }
```

#### 从控 `slave_controller.c` 修改要点
```c
// Init 中改为：先准备反馈包，再启动全双工 DMA
SlaveController_PrepareFeedbackPacket();
HAL_SPI_TransmitReceive_DMA(&hspi1,
                            (uint8_t*)&g_slaveCtrl.spiFeedbackPacket,
                            (uint8_t*)&g_slaveCtrl.spiRxPacket,
                            spi_length);

// Process 中：RX/TX 同时完成，只保留一个分支
if (g_slaveCtrl.spiRxComplete) {
    g_slaveCtrl.spiRxComplete = 0;
    g_slaveCtrl.spiTxComplete = 0;
    SlaveController_ProcessSPIPacket();
    SlaveController_PrepareFeedbackPacket();  // 准备下一轮反馈
    HAL_SPI_TransmitReceive_DMA(&hspi1, ...);
}

// 回调合并为：
void HAL_SPI_TxRxCpltCallback(SPI_HandleTypeDef *hspi) {
    SlaveController_SPI_RxCpltCallback();
    SlaveController_SPI_TxCpltCallback();
}
```

### 经验教训
- **全双工 SPI 不能只用 `HAL_SPI_Transmit_DMA`**：MISO 线上的数据会默默累积到主控 RX FIFO，很快触发 FIFO overrun 让硬件停摆。
- 一旦发现从控 CNDTR 停在某个固定值（且正好等于 FIFO 深度 × 对齐宽度），优先怀疑是 FIFO 阻塞/overrun。
- 降频只能让问题晚出现一点，不能根治——根因在硬件 FIFO 满而不是时序。
- 顺便：主控新增的 `g_spi_rx_buffer` 可以用来做 SPI 心跳/反馈校验，后续无需重构。
